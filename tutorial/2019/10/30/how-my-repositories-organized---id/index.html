    Bagaimana Aku Menangani Repositories di Android - Part #1

    Tutorial
    October 30, 2019 Aditya Chandra

    Pembuka

    Arsitektur dalam pengembangan aplikasi itu penting, biar ngga keteteran sana sini ketika ngoding, apalagi maintenance. Tentu kita ga bisa stuck dengan cara ngoding tradisional tanpa arsitektur. Ha?

    Ya maksudnya, kita punya puluhan bahkan ratusan method di dalam satu activity.

    Misalnya kita mau open connection ke SQLite, kita copas dari class activity yang sudah nerapin kodingan tersebut, karena kita ga mau susah. Atau, kita lakukan penarikan data SQLite dan membuat request API di dalam activity.

    ... DI DALAM ACTIVITY WOI!

    "Tapi kan, data-nya emang dibutuhkan sama Activity itu."

    Hm.

    Ya emang sih. TAPI YA, itu bikin NGERI. Semisal kamu punya 50 Activity, kemudian ada update di 1 API, kamu mau ganti satu - satu ngliatin activity-nya? Ngga deh, mending yuk move-on!

    Minimal, gunakan repository pattern.

    "Ha? Github donk. Pake repository."

    Bukan WOI.

    Tapi, okelah begini penjelasannya.

    Tentang Repository

    Repository berguna sebagai single source of truth atas semua data yang kamu punya. Ini adalah ilustrasi yang menggambarkannya dengan baik, lihat :

    Gambar di atas sudah menjelaskan cukup baik tentang konsep single source of truth dari repository, dimana repository menjadi satu - satunya sumber atas semua data yang diambil, dari API maupun SQLite.

    Maka, apabila LoginActivity membutuhkan proses login dari API, maka LoginActivity hanya perlu melaksanakan proses login melalui repository. Begitu pula, apabila HomeActivity memerlukan data profil pengguna, berita ataupun list banner, maka HomeActivity tersebut perlu meminta melalui repository juga.

    Sayangnya, itu hanya SATU repository.

    Pada penerapan repository, kita bisa membuat 1 singleton class untuk mengatur semua aliran data. TAPI YA, namanya singleton, kalo makin banyak Endpoint API dan table SQLite yang dibuat, pasti semakin gemuk dan tidak beraturan jadinya.

    Oleh karena itu, diperlukan klasifikasi repository yang lebih spesifik, seperti ini :

    Di sini kita membagi repository menjadi 3 tugas spesifik :

    1. User Repository - Dimana UserRepository bertanggungjawab atas semua data yang berhubungan dengan user. Seperti proses pendaftaran, autentikasi, pengambilan profil ataupun melakukan perubahan profil.

    2. News Repository - Bertanggungjawab atas semua data yang berhubungan dengan berita. Seperti pengambilan data berita, penyaringan (filter) berita, berita populer ataupun proses pemberian komentar / like kepada berita tertentu.

    3. Content Repository - Bertanggungjawab atas data yang berhubungan dengan konten atau preferensi singkat. Seperti list banner, notifikasi, penggunaan tema dark / light dan semacamnya.

    Penentuan jenis repository ini terserah kalian. Sesuaikan saja dengan kondisi dan kebutuhan.

    Meskipun kita membagi repository menjadi 3 bagian, BUKAN BERARTI bahwa SQLite dan API-nya harus ada 3. Ilustrasinya seperti ini :

    Dari ilustrasi di atas, akses API hanya ada 1, namun API memiliki beberapa endpoints seperti :

    • login [POST /api/v1/login]
    • register [POST /api/v1/register]
    • profile [GET /api/v1/me]
    • allnews [GET /api/v1/news]
    • filternews [GET /api/v1/news?query=search&sort=relevance]
    • popularnews [GET /api/v1/news?by=popular]
    • bannerlist [GET /api/v1/banners]
    • dan lainnya.

    Kemudian untuk setiap endpoints kita menentukan tipe repository-nya :

    • login [POST /api/v1/login] - UserRepo
    • register [POST /api/v1/register] - UserRepo
    • profile [GET /api/v1/me] - UserRepo
    • allnews [GET /api/v1/news] - NewsRepo
    • filternews [GET /api/v1/news?query=search&sort=relevance] - NewsRepo
    • popularnews [GET /api/v1/news?by=popular] - NewsRepo
    • bannerlist [GET /api/v1/banners] - ContentRepo

    Mudah, kan?

    Perhatikan juga, bahwa ilustrasi ini menunjukkan bahwa :

    • LoginActivity yang bertugas sebagai halaman masuk pengguna, hanya menggunakan UserRepository.

    • RegisterActivity yang bertugas sebagai halaman pendaftaran pengguna, hanya menggunakan UserRepository juga.

    • HomeActivity yang menampilkan Banner, berita populer, dan sedikit informasi pengguna menggunakan UserRepository, NewsRepository dan ContentRepository secara bersamaan.

    • NewsActivity yang menampilkan dan memberikan fitur pencarian berita hanya menggunakan NewsRepository saja.

    Maka, penggunaan repository dilakukan kepada activity yang membutuhkan saja.

    Misalnya, LoginActivity sangat tidak perlu menggunakan NewsRepository, karena tidak ada satupun informasi tentang berita yang ditampilkan di LoginActivity. Berlaku pula pada RegisterActivity, toh jika memang ada berita yang tampil di halaman Login dan Registrasi, maka kamu bisa gunakan NewsRepository.
